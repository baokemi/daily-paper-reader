## 摘要
现有的合成工具使用语料库主要用于离线监督微调，而强化学习（RL）则需要可执行的环境，以支持基于奖励检验的在线 rollout。我们提出了 COVERT，这是一种两阶段流水线，首先通过自进化合成和多层验证生成可靠的基础工具使用轨迹，然后施加保留 oracle 的增强操作，系统性地提高环境复杂度。这些增强引入了干扰工具、间接或模糊的用户查询，以及带有噪声、多格式或错误的工具输出，同时严格保留 oracle 工具调用和最终答案作为真实答案（ground truth）。该设计允许通过参考匹配对标准情况进行自动奖励计算，并通过轻量的审查辅助验证来处理诸如错误检测等特殊行为，从而支持工具调用策略的 RL 优化。在 Qwen2.5-Instruct-14B 模型上，COVERT-RL 使在 BFCL v3 上的总体准确率从 56.5 提升至 59.9，在 ACEBench 上从 53.0 提升至 59.3，且在通用能力基准上几乎没有退化；当与 SFT 结合使用时，进一步分别达到 62.1 和 61.8，验证了增益的叠加效果。这些结果表明，保留 oracle 的合成环境提供了一种可行的 RL 微调阶段，可作为 SFT 的补充，用于在存在歧义和不可靠工具反馈的情况下提高工具使用的鲁棒性。

## Abstract
Existing synthetic tool-use corpora are primarily designed for offline supervised fine-tuning, yet reinforcement learning (RL) requires executable environments that support reward-checkable online rollouts. We propose COVERT, a two-stage pipeline that first generates reliable base tool-use trajectories through self-evolving synthesis with multi-level validation, and then applies oracle-preserving augmentations that systematically increase environmental complexity. These augmentations introduce distractor tools, indirect or ambiguous user queries, and noisy, multi-format, or erroneous tool outputs, while strictly preserving oracle tool calls and final answers as ground truth. This design enables automatic reward computation via reference matching for standard cases and lightweight judge-assisted verification for special behaviors such as error detection, supporting RL optimization of tool-calling policies. On Qwen2.5-Instruct-14B, COVERT-RL improves overall accuracy on BFCL v3 from 56.5 to 59.9 and on ACEBench from 53.0 to 59.3, with minimal regressions on general-ability benchmarks; when stacked on SFT, it further reaches 62.1 and 61.8, confirming additive gains. These results suggest that oracle-preserving synthetic environments offer a practical RL refinement stage, complementary to SFT, for improving tool-use robustness under ambiguity and unreliable tool feedback.

## 二、COVERT 方法论

### 核心思想  
提出一个两阶段的合成数据生成流水线 **COVERT（COntrollable and VERifiable Tool-use data synthesis）**，通过  
1. **可靠轨迹生成（Stage I）** —— 生成高质量、无噪声、单工具的基础调用数据；  
2. **保真增强（Stage II）** —— 在不破坏原始真值（oracle）的前提下添加可控扰动，用于 RL 奖励计算。  

目标是得到一个既能自动奖励判定、又能在训练中引入歧义、错误和复杂性的可控环境。

### 阶段一：可靠基础轨迹生成
- **输入定义**：\((s,u,T,c^*,o^*,a^*)\)，表示系统提示、用户请求、工具集、oracle 工具调用、输出与最终答案。  
- **自进化生成（self-evolving synthesis）**：从少量人工种子开始，LLM生成候选轨迹，经验证通过后再加入生成池，实现自举式扩展。  
- **多级验证机制**：  
  - 格式检查（Format Checker）：确保标签、JSON结构正确；  
  - 规则检查（Rule Checker）：确认参数类型、范围匹配；  
  - 语义验证（Semantic Checker）：由LLM判断调用是否合理、输出一致。  
- 生成过程随机抽样任务领域、参数类型和工具定义，提升多样性并防止模式坍塌。

### 阶段二：保真增强（oracle-preserving augmentation）
- 在保持 \(c^*, o^*, a^*\) 等真实字段的前提下，修改环境组件（系统提示、用户请求、工具输出）。  
- 六大增强类型（详见表1）：  
  1. **干扰工具（distractor tools）** — 向工具集中加入相似但无用函数；  
  2. **查询重写（query rewriting）** — 间接或模糊化用户输入；  
  3. **多格式 / 噪声输出（multi-format/noisy output）** — 改变输出格式或嵌入日志；  
  4. **错误输出（erroneous outputs）** — 模拟失败或错误返回；  
  5. **问题查询（problematic queries）** — 无关或参数缺失请求；  
  6. **其他上下文扰动** — 修改系统提示，制造误导信息。  
- 增强实例带有元标签 \(m_k\)，区分两类验证器：  
  - **Exact-verifiable**：保持 \(c^*, a^*\)，可直接匹配计算奖励；  
  - **Judge-assisted**：由轻量 LLM 评估响应是否符合能力目标（如错误检测）。  

### 奖励与训练机制  
- 奖励函数：  
  \[
  R(\tau)=\lambda_{fmt}r_{fmt}(\tau)+\lambda_{cor}r_{cor}(\tau,m_k)
  \]
  - \(r_{fmt}\)：格式合规（XML标签完整、JSON有效）；  
  - \(r_{cor}\)：正确性奖励，依赖 capability 标签，采用匹配或 LLM judge 判断。  
- **RL 算法：GRPO（Group-wise Relative Policy Optimization）**  
  - 无需显式价值函数，适合可验证奖励场景。  
  - 模型与合成环境交互，如调用错误则返回通用错误信息直到结束。
